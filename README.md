# Nodeorb - Логистическая Экосистема 

## Описание проекта
Nodeorb - это современная логистическая экосистема, поддерживающая коммерческие операции Система интегрирует беспилотные транспортные средства, дроны, автоматизацию складов и комплексное соответствие требованиям безопасности.

## Архитектура

### Микросервисная архитектура
- Каждый сервис развертывается независимо
- Использование Kubernetes для оркестрации
- Канареечные развертывания для минимизации рисков
- Коммуникация через REST/gRPC и шину событий Kafka

## Технологический стек
- **Бекенд**: Kotlin/Java для микросервисов
- **Кроссплатформенность**: Kotlin Multiplatform для мобильных/десктоп приложений
- **Фронтенд**: React/Vue для веб-дашбордов
- **Инфраструктура**: Kubernetes, Docker, GovCloud (AWS/Azure)
- **Очередь сообщений**: Apache Kafka
- **Базы данных**: PostgreSQL, PostGIS для геопространственных данных
- **Интеграция**: REST, gRPC, EDI (EDIFACT, XML)

## Основные сервисы

### 1. admin-backend - Административная панель (бекенд)
Управление пользователями, мониторинг системы, настройки

### 2. admin-frontend - Административная панель (фронтенд)
Кроссплатформенное приложение для администрирования

### 3. autonomous-ops - Автономные операции
AI/ML для автоматического принятия решений в логистике

### 4. customs-service - Таможенный сервис
Интеграция с таможенными органами и оформление деклараций

### 5. fms-service - Управление флотом
Мониторинг транспорта в реальном времени, IoT интеграция

### 6. freight-marketplace - Грузовая биржа
Blockchain-платформа для сопоставления грузов и транспорта

### 7. oms-service - Управление заказами
Полный жизненный цикл заказов в логистике

### 8. scm-service - Безопасность и комплаенс
Централизованное управление безопасностью и аудитом

### 9. tms-service - Управление транспортом
Планирование маршрутов с учетом геоограничений

### 10. wms-service - Управление складом
Складские операции с YMS и обратной логистикой

## Сборка и запуск

### Собрать все сервисы
```bash
./gradlew build
```

### Запустить конкретный сервис
```bash
cd service-name && ./gradlew bootRun
```

### Запуск в Docker

#### Последовательный запуск с профилями

Создан новый оптимизированный стек с профилями и проверкой готовности контейнеров:

```bash
# Запустить только инфраструктуру (Postgres, Redis, Keycloak, EMQX, Elasticsearch, Kibana, Grafana, Prometheus, Zipkin)
docker-compose --profile infra up -d

# Запустить только сервисы (после готовности инфраструктуры)
docker-compose --profile services up -d

# Запустить только Freight Marketplace
docker-compose --profile marketplace up -d

# Запустить полный стек сразу
docker-compose --profile all up -d

# Проверить статус контейнеров
docker-compose ps

# Просмотреть логи
docker-compose logs -f

# Остановить стек
docker-compose down
```

#### Доступные профили

| Профиль          | Содержимое                                                                 |
|-------------------|-----------------------------------------------------------------------------|
| `infra`           | Postgres (с PostGIS), Redis, Keycloak, pgAdmin, EMQX, Elasticsearch, Kibana, Grafana, Prometheus, Zipkin |
| `services`        | SCM, OMS, WMS, FMS, TMS, Autonomous Ops, Customs Service, Admin Backend    |
| `marketplace`     | Freight Marketplace с отдельной БД Postgres                                  |
| `all`             | Все сервисы и инфраструктура (инфра + сервисы + marketplace)                 |

#### Пример запуска

```bash
# Запустить инфраструктуру и дождаться готовности
docker-compose --profile infra up -d

# Проверить статус инфраструктуры
docker-compose ps

# Запустить сервисы
docker-compose --profile services up -d
```

#### Freight Marketplace отдельно

```bash
# Запустить только Freight Marketplace
docker-compose --profile marketplace up -d

# Проверить статус
docker-compose ps
```

## Требования к безопасности
- Все конечные точки требуют аутентификации и авторизации
- Реализация шифрования FIPS 140-2
- Использование mTLS для сервис-сервис коммуникации
- Принцип наименьших привилегий
- Логирование всех попыток доступа и изменений данных

## Соответствие стандартам
- **US**: FedRAMP, CMMC Level 3, NIST 800-171, DFARS, ITAR/EAR
- **EU**: GDPR, CSRD, Digital Transport Law, Ecodesign Directive
- Реализация DLP (Data Loss Prevention)
- Поддержка CUI (Controlled Unclassified Information)

## Режимы работы
### 1. Коммерческий режим
Стандартные коммерческие операции

### 2. Режим стратегической устойчивости
Правительственные/аварийные операции с приоритетом

### Переключение режимов
- UI четко показывает текущий режим
- Активация стратегического режима требует повышенных привилегий
- Стратегический режим может переопределять коммерческие операции
